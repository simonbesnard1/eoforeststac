00
Pipeline Overview
How ingestion works
01
alsdb / ULS R pipeline produces a Zarr v3 store with resolution groups (1m/, 10m/, 100m/) containing gridded LiDAR products.
02
Convert the Zarr v3 store to Zarr v2 using the conversion script. This produces one standalone .zarr per resolution.
03
Register the region in the product config (REGIONS dict) with bbox, CRS, time period, and a Zarr name prefix.
04
Run the writer to process the Zarr v2 stores (rechunk, add metadata, set CRS) and upload them to S3.
05
Rebuild the catalog to update the STAC JSON on S3. The new items appear in the STAC Browser automatically.

Repository Layout
Project structure
📂
Each product type (ALS, ULS) has three modules: a product config (variables, regions, metadata), a catalog module (STAC collection + items), and a writer (load, process, write to S3).

02
Product Config
Add a new region / campaign
①
Open the product config: products/als_products.py for ALS or products/uls_products.py for ULS.
②
Add a new entry to the REGIONS dict. You need: a unique key, a human label, a scientific description, the bounding box in decimal degrees [west, south, east, north], a GeoJSON polygon geometry, the EPSG code, start/end dates, and a Zarr name prefix.
③
That's it. The catalog module automatically creates a STAC item with assets for every resolution (1m, 10m, 100m).
products/als_products.py
Adding a region 
REGIONS = {
    "spain_pnoa": { ... },  # existing

    # Add your new region here:
    "my_new_campaign": {
        "label": "My Campaign Name",
        "description": (
            "ALS point cloud products derived from ..."
        ),
        "bbox": [10.4, 51.0, 10.5, 51.1],  # [W, S, E, N] in degrees
        "geometry": {
            "type": "Polygon",
            "coordinates": [[
                [10.4, 51.0], [10.4, 51.1],
                [10.5, 51.1], [10.5, 51.0],
                [10.4, 51.0],
            ]],
        },
        "proj_epsg": 32632,         # native CRS of the data
        "start_datetime": datetime.datetime(2022, 1, 1, tzinfo=datetime.timezone.utc),
        "end_datetime": datetime.datetime(2022, 12, 31, tzinfo=datetime.timezone.utc),
        "zarr_name": "ALS_MY_CAMPAIGN",  # prefix for S3 Zarr filenames
    },
}
The same pattern works for ULS — just edit products/uls_products.py instead. The variables and resolutions are shared; only the region metadata differs.
03
Zarr Conversion
Convert Zarr v3 to v2
⚠
The alsdb pipeline produces Zarr v3 stores, but eoforeststac uses Zarr v2. You must convert before ingesting. Run the conversion in the alsdb environment (which has zarr v3).
terminal
Shell 
# Activate the alsdb environment (has zarr v3)
mamba run -n alsdb python scripts/convert_alsdb_to_zarr2.py \
    /path/to/alsdb/output/my_campaign/products.zarr \
    /path/to/output/my_campaign/
📁
This produces three Zarr v2 stores: 1m.zarr, 10m.zarr, 100m.zarr — each a standalone xarray-readable dataset.
For ULS data produced by Linda's R pipeline: if the output is already Zarr v2, skip this step and point the writer directly at the Zarr stores. If it's Zarr v3, use the same conversion script.
04
Writer
Push data to S3
①
The writer reads each Zarr v2 store, applies metadata (CRS, variable attributes, fill values), rechunks for cloud-optimal access, and writes to the GFZ S3 bucket.
②
Use the write_region() method to process all three resolutions at once. The S3 credential profile is atlaseo-glm.
ingest_als.py
Python · ALS 
from eoforeststac.writers.als_products import ALSProductsWriter

writer = ALSProductsWriter(
    endpoint_url="https://s3.gfz-potsdam.de",
    bucket="dog.atlaseo-glm.eo-gridded-data",
    profile="atlaseo-glm",
)

# All three resolutions at once:
writer.write_region(
    "/path/to/output/my_campaign/",
    region="my_new_campaign",  # must match REGIONS key
    version="1.0",
)
ingest_uls.py
Python · ULS 
from eoforeststac.writers.uls_products import ULSProductsWriter

writer = ULSProductsWriter(
    endpoint_url="https://s3.gfz-potsdam.de",
    bucket="dog.atlaseo-glm.eo-gridded-data",
    profile="atlaseo-glm",
)

writer.write_region(
    "/path/to/uls/hainich/",
    region="hainich",
    version="1.0",
)
The region parameter must exactly match a key in the REGIONS dict. The writer expects the input directory to contain 1m.zarr, 10m.zarr, and 100m.zarr.
05
STAC Catalog
Rebuild the catalog
①
After uploading data to S3, rebuild the STAC catalog so the new items appear in the browser. This writes updated JSON to both the internal and public S3 paths.
terminal
Shell 
mamba run -n eoforeststac python scripts/create_catalog.py
🔎
Check the result in the STAC Browser. Your new item should appear under the "ALS / ULS / LiDAR" theme with all resolution assets listed.
→ Open STAC Browser
06
Validation
Verify and test
①
Run the test suite to verify your new region passes all STAC validation checks, pydantic config validation, and consistency tests.
terminal
Shell 
mamba run -n eoforeststac python -m pytest eoforeststac/tests/test_stac_validation.py -v
②
Verify the data is readable from S3 by opening it with xarray. Use anonymous access — no credentials needed for reading.
verify_read.py
Python · Verify 
import xarray as xr
import s3fs

fs = s3fs.S3FileSystem(
    client_kwargs={"endpoint_url": "https://s3.gfz-potsdam.de"},
    anon=True,
)

store = s3fs.S3Map(
    root="dog.atlaseo-glm.eo-gridded-data/collections/ALS_PRODUCTS/ALS_MY_CAMPAIGN_10m_v1.0.zarr",
    s3=fs, check=False,
)
ds = xr.open_zarr(store)
print(ds)
③
Apply black formatting before committing: mamba run -n eoforeststac python -m black eoforeststac/
④
Commit and push. The CI workflow on GitHub will run the test suite automatically.
Checklist for a new campaign:
1. Region added to REGIONS in product config
2. Zarr v3 converted to v2
3. Writer pushed data to S3
4. Catalog rebuilt (scripts/create_catalog.py)
5. Tests pass
6. Data readable from S3 with anonymous access
7. Item visible in STAC Browser
8. Code formatted with black and committed
